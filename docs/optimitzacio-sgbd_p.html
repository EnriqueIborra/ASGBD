Monitoratge del Sistema Gestor de Bases de Dades (SGBD)

Què és el monitoratge?

El monitoratge és el procés de controlar el funcionament del sistema gestor, detectant possibles errors o situacions que puguin afectar el rendiment. Ha de ser poc intrusiu, per tal de no interferir en el funcionament normal del sistema.

S'ha de tractar que les tasques de monitoratge i diagnòstic del sistema siguen el menys intrusives possible perquè no penalitzen el rendiment del sistema gestor, i relegar les tasques que requerisquen un consum de recursos mitjà o elevat per a realitzar-les en moments de baixa càrrega

🧰 Eines principals de monitoratge

    Vistes del sistema
  • Activitat en temps real: pg_stat_activity
  • Estadistiques de taules: pg_stat_user_tables
  • Bloquejos: pg_locks
  • Consultes lentes: pg_stat_statements (extensió)
    Extensions de monitoratge
  • pg_stat_statements
  • auto_explain
  • pgstattuple
  • pgaudit
    Eines gràfiques
  • pgAdmin
  • Prometheus + Grafana
  • Zabbix / Nagios
    Altres
  • Logs del servidor
  • Plans d'execució

📝 Registre d’errors (fitxers de logs)

Registra activitats com arrencades, aturades, errors, consultes lentes, etc.

  • Error: problemes greus
  • Warning: advertències
  • Debug: informació detallada (en desenvolupament)

Optimització

🎯 Objectiu

L’objectiu de l’optimització és millorar el rendiment del sistema, tot reduint els temps de resposta, l’ús de recursos i l’impacte en l’usuari final. No es tracta només de fer que les consultes funcionen, sinó que siguen ràpides i eficients.

Nivells on optimitzar

    Entorn
    • a nivell de Sistema Operatiu
    • a nivell de Xarxa
    Sistema Gestor
    • Grandària i ubicació de fitxers de dades
    • Grandària de l'emmagatzemament temporal
    Base de Dades
    • Disseny de taules i tipus de dades (Ajustar al necessari)
    • Camps calculats (intentar mantindre els menys possibles)
    • Desnormalització (reduir JOIN a costa de augmentar redundància)
    • Desfragmentació: VACUUM, VACUUM FULL, AUTOVACUUM
    • Particionament
    • Crear, modificar o eliminar índex
    • Optimització de consultes

Particionament

  • S’evita processar tota una taula (sols es processa la partició)
  • Les dades poden ser accedides en paral·lel
  • Facilita operacions com p.e. el purgat de dades

    • Creació d'índex

    • Sobre que columnes crear
      • Claus primaries i alienes ( normalment ja ho fa el SGBD)
      • Columnes que habitualment apareixen en SELECT i en WHERE
      • Columnes amb bona selectivitat ( si poques files tenen el mateix valor)
    • Sobre que columnes NO crear
      • Taules amb poques dades
      • Columnes amb molts valors NULL
      • Columnes amb valors que es modifiquen molt sovint
      • Índex sobre moltes columnes
      • Molts índex per taula
    • Que tipus d’índex crear
      • Índex B-tree
      • GIN
      • GiST
      • BRIN
    s

    Optimització de consultes

    • Usar taules derivades, subconsultes i joins
    • Cursors i funcions

    Pràctiques recomanades per a índex

    • Columna freqüentment filtrada (WHERE)
    • Columnes utilitzades en JOINs
    • ⚠️ No crear índexs per defecte a totes les columnes (consumeixen espai i poden alentir INSERT/UPDATE)

    Pràctiques recomanades per a consultes

    • Utilitza EXPLAIN per analitzar l'estratègia de consulta (s'explica en el següent punt)
    • Evita SELECT * : només demana les columnes necessàries
    • Evita subconsultes correlacionades no indexades
    • Fes ús de joins eficients (preferentment explícits)
    • Controla l’ús de funcions sobre columnes indexades

    🛑 Consultes ineficients

    Exemples a evitar:

    • SELECT * FROM taula; → molt costós si té moltes columnes
    • WHERE TO_CHAR(data, 'YYYY') = '2023' → trenca índexs
    • WHERE UPPER(nom) = 'JOAN' → si no hi ha índex amb UPPER

    Versions millorades

    -- Evitar funcions sobre columnes
WHERE data >= TO_DATE('01/01/2023', 'DD/MM/YYYY')

-- Substituir subconsulta per JOIN
SELECT a.nom, c.nom_curs
FROM alumnes a
JOIN cursos c ON a.curs_id = c.id;

    Altres consells d’optimització

    • Usa BULK COLLECT i FORALL per treballar amb grans volums en PL/SQL
    • Utilitza particions en taules molt grans
    • Evita bloquejos i transaccions llargues

    Resum de bones pràctiques

    • Coneix el model de dades i el volum
    • Usa EXPLAIN sovint
    • Mantén estadístiques al dia
    • Indexa amb criteri
    • Millora consultes repetitives o lentes

    Optimització en PostgreSQL

    📈 L'optimitzador

    Postgres utilitza un component anomenat query planner/ optimizer per determinar el millor pla d’execució per a una consulta SQL. L’objectiu és minimitzar el temps d’execució i l’ús de recursos.

    📊 Estadístiques

    Postgres utilitza un optimitzador basat en costos anomenat "planner"

    L’optimitzador utilitza estadístiques sobre les taules, columnes i índexs per decidir l’estratègia d’execució.

    Com generar estadístiques

    -- Es fan automaticament, però es poden fer manualment amb
    ANALYZE;
    ANALYZE alumnes;
    ANALYZE alumnes(edat, nom);
    VACUUM ANALYZE alumnes;

    ⚠️ és molt importat crear tasques programades que realitzen esta actualització fora d'hores de càrrega de treball

    Com vore estadístiques

    SELECT *  FROM pg_stats  WHERE tablename = 'alumnes';

    També es poden automatitzar mitjançant tasques de manteniment o recollida automàtica.

    🧪 EXPLAIN

    Aquesta ordre permet visualitzar el pla que Postgres seguirà per executar una consulta (sense executar-la realment). Mostra si s’utilitzen índexs, escaneigs complets, joins, etc. Ajuda a detectar consultes lentes o mal optimitzades.

      S'usa principalment per:
    • Analitzar rendiment
    • Detectar colls de botella
    • Veure si s'estan usant index
    • Entendre perquè és lenta una consulta
    • Ajuda a optimitzar una SQL

    Exemple d’ús

    EXPLAIN SELECT * FROM alumnes WHERE edat > 18;
o 
EXPLAIN ANALYZE SELECT * FROM alumnes WHERE edat > 18;

    EXPLAIN a pgAdmin (PostgreSQL)

      Usant el botó Explain
    • Escriu la consulta al query tool (worksheet)
    • Selecciona la consulta o deixa el cursor dins
    • Fes clic al botó Explain (icona amb lupa o arbre)
    • Mostra el pla estimat sense executar la consulta
    • Opcionalment, fes clic a Explain Analyze per veure el pla real amb temps d’execució i files retornades

    Manualment amb SQL

    -- Pla estimat
EXPLAIN SELECT * FROM alumnes WHERE edat > 18;

-- Pla amb execució real
EXPLAIN ANALYZE SELECT * FROM alumnes WHERE edat > 18;

    Factors que afecten el pla d’execució

    • Disponibilitat d’índexs
    • Estadístiques actualitzades
    • Volum de dades
    • Condicions de filtre i joins

    Recomanacions generals

    • Revisa les consultes lentes amb EXPLAIN
    • ANALYZE taula; EXPLAIN ANALYZE;
    • Actualitza estadístiques sovint
    • Fes servir índexs de manera estratègica
    • Prova diferents opcions de joins segons el volum i condicions
    • Evita accessos complets si pots usar índexs

    Exemple de consulta optimitzada

    Consulta original (lenta):

    SELECT * FROM alumnes WHERE TO_CHAR(data_naixement, 'YYYY') = '2005';

    Versió optimitzada:

    SELECT * FROM alumnes
WHERE data_naixement >= DATE '2005-01-01'
  AND data_naixement < DATE '2006-01-01';

    💬 Conclusió

    L’optimització en PostgreSQL es basa en la col·laboració entre el desenvolupador i l’optimitzador del sistema. Mantenir estadístiques actualitzades, evitar males pràctiques i revisar plans d’execució són claus per un rendiment òptim.

    Eines d'optimització en PostgreSQL

    🎯 Objectiu

    Aquestes eines permeten analitzar el rendiment de consultes SQL, sessions i càrregues del sistema, ajudant a diagnosticar colls d’ampolla i a optimitzar els accessos a dades.

    1. EXPLAIN

    Mostra el pla d’execució previst per postgresql per a una consulta. Ajuda a detectar si s’utilitzen índexs o si es fa un escaneig complet de taula. 

    EXPLAIN; EXPLAIN ANALYZE;  
EXPLAIN SELECT * FROM alumnes WHERE edat > 18;

-- amb execució real
EXPLAIN ANALYZE SELECT * FROM alumnes WHERE edat > 18;

    Sortides habituals:

      Mostra:
    • cost estimat
    • temps d’execució (amb ANALYZE)
    • joins (Nested Loop, Hash Join, Merge Join)

    2. pg_stat_statements

    Extension que guarda estadístiques de totes les consultes executades. Permet identificar les consultes més pesades i el seu impacte en CPU, I/O i temps.

    -- Top 10 consultes per temps total
SELECT query, calls, total_time, rows
FROM pg_stat_statements
ORDER BY total_time DESC
LIMIT 10;

    3. pg_stat_activity

    Permet veure l’estat de les sessions actives i quines consultes s’estan executant: 

    SELECT pid, usename, state, query, query_start
FROM pg_stat_activity
WHERE state = 'active';
    • Útil per detectar sessions que bloquegen recursos o provoquen càrrega elevada
    • Combina amb pg_locks per veure bloquejos:
    SELECT * FROM pg_locks;

    4. Estadístiques de taula

    El planner de PostgreSQL utilitza estadístiques per decidir el millor pla. ANALYZE → actualitza estadístiques d’una taula o columnes concretes:

    ANALYZE alumnes;
ANALYZE alumnes(edat, nom);

    VACUUM ANALYZE → neteja + actualització de estadístiques:

    VACUUM ANALYZE alumnes;

    AUTOVACUUM → automàtic, manté estadístiques actualitzades sense intervenció manual

    5. Altres eines gràfiques

    • pgAdmin Dashboard → monitoratge de sessions, càrrega i consulta de plans
    • pgBadger → informes d’ús i rendiment basats en logs
    • Grafana + Prometheus → dashboards de rendiment avançats

    Bones pràctiques d’optimització de consultes

    • Evita SELECT *, només selecciona les columnes necessàries
    • Indexa columnes que apareixen en WHERE i JOIN
    • Evita subconsultes correlacionades no indexades
    • Avalua l’ús de EXPLAIN ANALYZE abans i després de canvis
    • Prova diferents tipus de joins segons volums de dades (Nested Loop, Hash Join)

    Fragmentació de Taules en PostgreSQL

    🔍 Què és la fragmentació de taules?

    La fragmentació (o bloat) fa referència a l'espai de memòria mal aprofitat dins d'una taula (o índex) a la base de dades. Pot provocar que certes operacions, com escaneigs complets de taula o accés via índex, siguin menys eficients.

    Causes comunes de fragmentació en PostgreSQL

    • Insercions i esborrats freqüents: deixen "forats" dins dels blocs que no es reomplen automàticament.
    • Actualitzacions amb valors més grans: poden generar row chaining.

    Conseqüències de la fragmentació

    • Rendiment més lent en consultes.
    • Increment d'I/O.

    🧰 Com detectar la fragmentació

    -- mida aproximada de taula i índexs
SELECT 
    relname AS taula,
    pg_total_relation_size(relid) AS mida_total_bytes,
    pg_relation_size(relid) AS mida_taula_bytes,
    pg_indexes_size(relid) AS mida_indexes_bytes
FROM pg_stat_user_tables;

-- mida real dels blocs i tuples buides
SELECT *, 
       (pg_total_relation_size(relid) - pg_relation_size(relid)) AS bloat_bytes
FROM pg_stat_user_tables;

    ambé es poden usar extensions com pgstattuple:

    CREATE EXTENSION IF NOT EXISTS pgstattuple;
SELECT * FROM pgstattuple('alumnes');

    🛠️ Solucions per reduir o eliminar la fragmentació

    VACUUM;          -- neteja bàsica
VACUUM FULL;     -- recupera espai i compacta taula

    VACUUM → habitual i ràpid, manté estadístiques
    VACUUM FULL → lent, bloqueja la taula, recupera espai físic

    REINDEX TABLE alumnes;
-- o per un índex concret
REINDEX INDEX idx_nom;

    Recomanat quan els índexs tenen bloat

    CLUSTER alumnes USING idx_nom;

    Reorganitza físicament la taula segons un índex
    Millora accessos seqüencials
    Bloqueja la taula mentre s’executa

    CREATE TABLE alumnes_nova AS SELECT * FROM alumnes;
DROP TABLE alumnes;
ALTER TABLE alumnes_nova RENAME TO alumnes;

    Crear una taula nova i copiar dades:

    📊 Índexs

    Els índexs permeten accedir més ràpidament a dades. Cal usar-los correctament:

    🔧 Crear un índex

    CREATE INDEX idx_nom ON alumnes(nom);  
    create index nom_ind on nom_taula (camp1, camp2,..);
REINDEX INDEX nom_ind ;

    Quan es crea una taula amb una clau primaria o unique, PostgreSQL crea un índex automàticament

    Quan crees PRIMARY KEY o UNIQUE, PostgreSQL crea automàticament un índex B-tree

    Reconstruir índex

    REINDEX INDEX nom_ind ;

    Com explorar els índex (en DD)

    SELECT indexname, indexdef
FROM pg_indexes
WHERE tablename = 'alumnes';

    Particionament

    create table ....( ) partition by range(nomcamp) (....);

    En que consisteix el particionament?

    El particionament consisteix a dividir lògica i físicament una taula gran en parts més menudes anomenades particions, de manera que el sistema puga gestionar-les de forma més eficient.

    Per a què serveix?

    Per exemple imaginem una taula de factures, on tenim el detall de la nostra facturació al llarg de 9 anys, 2017, 2018… 2025, si volguérem fer:

    SELECT SUM(total_fac) FROM facturacio WHERE any = 2019;

    En aquest exemple s'hauria de recórrer tota la taula (imaginem que parlem de 30 milions de registres en total, és molt no?), per aquest motiu un criteri possible per a particionar la taula seria per l'any de la data de la factura

    • Millora del rendiment de consultes.
    • Facilita el manteniment (per exemple, eliminar o arxivar particions antigues).
    • Millor gestió de l'emmagatzematge.
    • Paral·lelisme en operacions.

    Tipus de particionament

    1. Particionament per rang (RANGE)

    Divideix segons un valor dins d’un rang (ex: dates).

    CREATE TABLE vendes (
    id SERIAL PRIMARY KEY,
    data_venda DATE,
    import NUMERIC
) PARTITION BY RANGE (data_venda);

-- Particions
CREATE TABLE vendes_2019 PARTITION OF vendes
    FOR VALUES FROM ('2019-01-01') TO ('2020-01-01');

CREATE TABLE vendes_2020 PARTITION OF vendes
    FOR VALUES FROM ('2020-01-01') TO ('2021-01-01');

CREATE TABLE vendes_2021 PARTITION OF vendes
    FOR VALUES FROM ('2021-01-01') TO ('2022-01-01');

CREATE TABLE vendes_future PARTITION OF vendes
    FOR VALUES FROM ('2022-01-01') TO ('9999-12-31');

    2. Particionament per llista (LIST)

    Divideix segons valors específics.

    
CREATE TABLE clients (
    id SERIAL PRIMARY KEY,
    nom VARCHAR(50),
    regio VARCHAR(20)
) PARTITION BY LIST (regio);

CREATE TABLE clients_nord PARTITION OF clients FOR VALUES IN ('NORD');
CREATE TABLE clients_sud PARTITION OF clients FOR VALUES IN ('SUD');
CREATE TABLE clients_est PARTITION OF clients FOR VALUES IN ('EST');
CREATE TABLE clients_oest PARTITION OF clients FOR VALUES IN ('OEST');

    Com saber quina estratègia escollir?

    Depèn de:

    • El tipus de consultes habituals.
    • La mida de les dades.
    • El tipus de columna (distribució, cardinalitat).
    • Necessitats de manteniment i arxiu.

    Altres operacions útils

    -- Afegir partició
CREATE TABLE vendes_2022 PARTITION OF vendes
    FOR VALUES FROM ('2022-01-01') TO ('2023-01-01');
  

-- Eliminar partició
DROP TABLE vendes_2020;

-- Consultar només una partició
SELECT * FROM vendes
WHERE data_venda >= '2021-01-01' AND data_venda < '2022-01-01';

    Consultar si una taula està particionada

    
SELECT relname AS table_name,
       relkind AS table_type
FROM pg_class
WHERE relname = 'vendes';    

o 

SELECT *
FROM pg_partitioned_table pt
JOIN pg_class c ON c.oid = pt.partrelid
WHERE c.relname = 'vendes';

    Optimització de consultes

    Pla d'execució

    PostgreSQL no emmagatzema els plans d’execució en una taula com Oracle, però el planner calcula un pla cada vegada que s’executa una consulta (SELECT, UPDATE, INSERT, DELETE).

      El pla indica:
    • Com es llegiran les dades (seqüencialment o amb índex)
    • Quin tipus de join s’utilitzarà (Nested Loop, Hash Join, Merge Join)
    • L’ordre d’accés a les taules
    • El cost estimat i files estimades

    Eines per vore el pla

    -- Pla estimat (no executa la consulta)
EXPLAIN SELECT * FROM t_pedidos WHERE codpedido = 5;

-- Pla amb execució real i temps real
EXPLAIN ANALYZE SELECT * FROM t_pedidos WHERE codpedido = 5;
  • PostgreSQL no requereix permisos especials per veure EXPLAIN
  • En pgAdmin, es pot fer clic a Explain o Explain Analyze per veure-lo gràficament

    • Estadistiques

    El planner utilitza estadístiques de taules i columnes (ANALYZE) per decidir el millor pla.

    -- Exemple per vore estadístiques
SELECT *
FROM pg_stats
WHERE tablename = 't_pedidos';

    Monitoratge de consultes

  • pg_stat_activity → sessions actives i queries en execució
  • pg_stat_statements → historial de consultes amb temps, I/O, número de crides
    • -- Consultes més lentes
SELECT query, calls, total_time, rows
FROM pg_stat_statements
ORDER BY total_time DESC
LIMIT 10;  
    -- Sessions actives
SELECT pid, usename, state, query, query_start
FROM pg_stat_activity
WHERE state = 'active';

    Insercions massives i manipulació de taules

    -- carregar dades massives des de fitxers CSV:
COPY t_pedidos FROM '/path/to/file.csv' CSV HEADER;  
    INSERT en bulk sense index temporalment 
  → opcional: treure temporalment indexes per a càrregues molt grans i reconstruir-los després amb REINDEX.

    TRUNCATE → eliminar totes les files ràpidament:

    TRUNCATE TABLE t_pedidos;

    Vistes materialitzades → PostgreSQL suporta MATERIALIZED VIEW:

    CREATE MATERIALIZED VIEW v_pedidos_totals AS
SELECT cliente_id, SUM(total_fac) AS total
FROM t_pedidos
GROUP BY cliente_id;

-- Actualitzar-la
REFRESH MATERIALIZED VIEW v_pedidos_totals;
